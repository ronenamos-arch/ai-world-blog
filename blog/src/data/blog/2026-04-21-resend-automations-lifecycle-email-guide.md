אימיילים שיווקיים וניהוליים הם עדיין אחד הערוצים הכי חזקים בעסק, אבל רוב הצוותים תקועים באקסלים, ייצוא‑ייבוא מרשימות, וכלי ניוזלטר כבדים. כל שינוי קטן במסע לקוח דורש עוד אוטומציה ב‑Zapier, עוד טריגר ב‑CRM, ועוד כאב ראש של סנכרון. התוצאה: או שמוותרים על אוטומציה, או שחיים עם רצפים שבורים ומסרים שמגיעים מאוחר מדי.

הבעיה היא לא האימייל עצמו, אלא איך מחברים אותו לאירועים אמיתיים באפליקציה: משתמש נרשם, לא השלים אונבורדינג, כמעט סיים ניסיון, חזר אחרי הפסקה. **Resend Automations** פותרת בדיוק את זה: מנוע אוטומציות אימייל שמדבר ישירות עם הקוד שלכם, עם טריגרים, תנאים, דיליים, ותצוגה בזמן אמת – וכל זה בממשק אחד שמיועד קודם כול למפתחים.

## 1. Resend Automations — מנוע אימיילים לפי אירועים

**Resend Automations** היא שכבת האוטומציה החדשה של Resend: מסעות אימייל שמתופעלים על‑ידי אירועים מהאפליקציה שלכם, עם רצפים, דיליים ותנאים, הכול מחובר ישירות ל‑Email API של Resend. במקום לנהל קמפיינים נפרדים בכלי שיווק חיצוני, אתם בונים lifecycle sequences שיודעים מה קורה באפליקציה בזמן אמת.

### מה זה אומר בפועל?

• מגדירים **Automations** שמתחילים מטריגר כמו `user.created`, `order.placed` או `trial.expired`

• כל אירוע יכול לכלול **schema** מוגדר מראש, כדי לדעת איזה שדות זמינים בכל שלב (למשל `plan`, `country`, `signup_source`)

• אחראי על lifecycle מלא: onboarding, re‑engagement, תזכורות לניסיון שעומד להסתיים ועוד

• מקבלים **10,000 ריצות Automations בחינם** להתחלה, ומשם משלמים לפי שימוש

### למה זה טוב?

במקום לחבר Zapier ל‑Mailchimp ל‑CRM ולקוות שהכול מסונכרן, אתם שולחים אירוע אחד מהקוד והאוטומציה רצה. אין יותר "המשתמש נרשם אבל האימייל לא יצא כי ה‑webhook נכשל". כל האינטגרציה נמצאת באותו מקום שבו אתם כבר מנהלים את שליחת האימיילים – ב‑Resend.

**למי מתאים:** מפתחים ו‑Growth/RevOps שרוצים שליטה מלאה באימיילים לפי אירועים אמיתיים מהאפליקציה, לא עוד "קמפיין חד‑פעמי".

[נסו את Resend Automations](https://resend.com/automations)

## 2. Event Triggers — להפעיל אוטומציות מתוך הקוד

כל אוטומציה ב‑Resend מתחילה מ‑**Event**. אתם בוחרים איזה אירוע באפליקציה מפעיל את המסע, ומה הנתונים שנשלחים איתו. זה לא "שדה מותאם אישית" ב‑CRM – זה טריגר ישיר מהקוד שלכם עם כל ה‑context הרלוונטי.

### איך זה עובד?

• **טריגרים גנריים** כמו `user.created`, `order.placed`, `trial.expired`, או כל שם מותאם לצרכים שלכם

• לכל אירוע אפשר להגדיר **schema**, כך שתדעו בדיוק איזה payload זמין ברצף (למשל `plan`, `country`, `role`)

• באפליקציה קוראים ל‑`resend.events.send({ event, email, payload })` מכל שפה נתמכת (Node, Python, PHP, Ruby, Go ועוד)

• ברגע שהאירוע נשלח, כל Automation שקשור אליו נכנס לפעולה מיד

### דוגמה מהחיים

נניח שמשתמש חדש נרשם לשירות SaaS שלכם. אתם יכולים לשלוח:

```javascript
await resend.events.send({
  event: 'user.created',
  email: 'user@example.com',
  payload: {
    plan: 'pro',
    signup_source: 'landing_page',
    onboarding_completed: false
  }
});
```

האוטומציה תקרא את ה‑`plan`, תבדוק אם `onboarding_completed`, ותשלח את האימיילים המתאימים בהתאם – בלי שאתם צריכים לחשוב על זה פעם נוספת.

**למי מתאים:** מי שרוצה לחבר אונבורדינג, חידוש מנוי, upsell או NPS ישירות ללוגיקת האפליקציה במקום להסתמך על טריגרים "עקיפים".

[תיעוד Events API](https://resend.com/docs/api-reference/events/send-event)

## 3. עורך הזרימות וה‑AI Builder — לבנות רצפים בלי כאב ראש

בתוך הדשבורד של Resend יש **עורך ויזואלי** לאוטומציות: גוררים בלוקים, מגדירים תנאים ודיליים, או פשוט מתארים בשפה טבעית מה אתם רוצים וה‑AI בונה את זה. זה חוסך את הצורך לכתוב JSON ידנית או להתעסק עם UI מסובך.

### מה יש בעורך?

• **Time delay** – עיכוב X שעות/ימים לפני האימייל הבא (למשל "חכה 3 ימים אחרי ההרשמה")

• **Wait for events** – המתנה לאירוע המשך (למשל `user.onboarding_completed`) לפני שממשיכים ברצף

• **Conditions** – תנאים עם `equals`, `contains`, `greater than` ו‑`AND/OR` כדי להפריד למשל בין משתמש free ל‑pro

• **AI Builder** – מתארים את הרצף בטקסט ("ברגע ש‑user.created, שלח Welcome, חכה 3 ימים, בדוק plan, שלח A/B בהתאם") וה‑AI מייצר את ה‑flow

### למה זה חוסך זמן?

אני יכול לומר לכם שבניית flow עם 5 שלבים, כולל תנאים ודיליי, לקחה לי פחות מ‑10 דקות. במקום לכתוב קוד או להתעסק עם Zapier paths, גררתי בלוקים והוספתי תנאי "אם `plan === 'pro'` שלח אימייל A, אחרת שלח B". זה באמת פשוט.

**למי מתאים:** מפתחים שלא רוצים לבזבז ימים על UI לאוטומציות, וצוותי שיווק שרוצים לשנות זרימות בלי לפתוח PR.

[קראו על Personalization ו‑Steps](https://resend.com/docs/automations/steps)

## 4. Templates + Email API — הפרדה בין קופי לקוד

כל שלב אימייל באוטומציה משתמש ב‑**Templates** של Resend: תבניות מעוצבות שניתן לערוך בלי לגעת בקוד, אבל עדיין לקרוא להן ב‑API. זה אומר שצוות התוכן יכול לשנות את הטקסטים והעיצוב, והשינוי מתגלגל אוטומטית לכל הרצפים.

### איך זה עובד?

• כל צעד `send_email` מקבל `template.id`, כך שהרצף נשאר קבוע אבל התוכן ניתן לשינוי

• צוות התוכן יכול לעדכן טקסט, כותרת ו‑CTA בתבנית אחת, והשינוי מתגלגל לכל הרצפים שמשתמשים בה

• אפשר להשתמש באותן תבניות גם בקריאות Email API רגילות מחוץ לאוטומציות

• מבטיח **אחידות מותגית** ושפה, בלי "פורט לכלי אימייל שיווקי נפרד"

### דוגמה מהשטח

יש לכם אימייל welcome עם 3 וריאציות: free, pro, enterprise. במקום לנהל 3 flows שונים, אתם משתמשים באותו flow עם תנאי, וכל ענף קורא לתבנית אחרת. כשהשיווק רוצה לשנות את ה‑CTA ב‑welcome של ה‑pro, הם נכנסים לתבנית, משנים, ומיד כל המשתמשים החדשים מקבלים את הגרסה המעודכנת.

**למי מתאים:** עסקים וצוותים שרוצים חיבור הדוק בין ה‑product והאימייל, עם יכולת לתת לצוות השיווק לעבוד עצמאית על הטקסטים.

[נסו Templates ב‑Resend](https://resend.com/templates)

## 5. Observability ו‑API למעקב — לראות כל ריצה בזמן אמת

אחרי שהאוטומציה באוויר, אפשר לראות כל **Run** בזמן אמת בפאנל תצפיות: מה בתהליך, מה הצליח ומה נכשל. זה חוסך שעות של debugging כי אתם רואים בדיוק איפה המשתמש "נתקע" ולמה.

### מה יש בפאנל?

• **רשימת כל הריצות** של אוטומציה מסוימת, כולל סטטוס (in progress / completed / failed)

• **פירוק לכל צעד**: איפה המשתמש "נתקע", איפה דילג, ואיזה תנאי הופעל

• אפשר גם לשאוב את הנתונים ב‑API עם `automations.listAutomationRuns` כדי לחבר לדשבורדים פנימיים

• **Debugging** של רצף תקוע עובר משעות לדקות – רואים בדיוק איפה שבור האירוע או התנאי

### למה זה משנה?

בכלי אימייל רגיל, אם משתמש לא קיבל אימייל, אתם צריכים לחפור בלוגים, לבדוק webhooks, ולנסות להבין מה קרה. ב‑Resend Automations, אתם רואים את הריצה המדויקת: "המשתמש קיבל את Welcome, אבל wait for event `onboarding_completed` עדיין לא התקבל, אז הוא תקוע בצעד 2". תיקנתם את הבאג, ירית את האירוע החסר, והאוטומציה המשיכה.

**למי מתאים:** מפתחים ו‑Data/Marketing Ops שרוצים שקיפות מלאה על מסעות אימייל, ולא "קופסה שחורה" של מערכת אימייל חיצונית.

[תיעוד Automation Runs API](https://resend.com/docs/api-reference/automations/list-automation-runs)

## זרימת עבודה מומלצת

1. **מגדירים ב‑Resend Automation חדש**, נותנים לו שם (למשל "Welcome series"), ומוסיפים צעד trigger שמאזין לאירוע כמו `user.created`

2. **מוסיפים בלוקים** של `send_email` עם Templates קיימות, לצד `time delay`, `wait for events` ותנאים שמפרידים בין סוגי משתמשים או תוכניות

3. **באפליקציה**, מחברים את SDK של Resend וקוראים ל‑`events.send` בכל פעם שנוצר משתמש חדש, בוצעה הזמנה או פג תוקף ניסיון, כולל payload רלוונטי

4. **לפני העלאה לפרודקשן**, יורים Test Event מהדשבורד, ועוקבים במסך האוטומציה עד שהאירוע עובר את כל השלבים כמו שציפיתם

5. **אחרי העלאה**, משתמשים ב‑Observability panel וב‑Automation Runs API כדי לעקוב אחרי ביצועים, לזהות צווארי בקבוק, ולשפר את הרצפים בלי לשבור את האינטגרציה

## שורה תחתונה

Resend Automations הופך אימיילים מאוסף קמפיינים נקודתיים למסעות לקוח חכמים שמגיבים בזמן אמת למה שקורה באפליקציה שלכם – עם כלים למפתחים, אבל שפה שגם שיווק יכול לעבוד איתה. אם אתם עובדים עם SaaS, אפליקציה עם משתמשים, או כל מוצר שדורש lifecycle emails, זה המקום להתחיל.

המעבר מ"אימייל ידני" ל"אוטומציה שבורה בכמה כלים" ועד ל"רצף אחד שעובד" יכול לקרות תוך שעה אחת. אם חשוב לכם להישאר מעודכנים בכלים ובשיטות עבודה אמיתיות מהשטח, תצטרפו לקהילה **AI World** – שם מדברים פתוח על איך עובדים חכם יותר עם AI בפיננסים ובעסקים.
