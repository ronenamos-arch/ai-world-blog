רוב האנשים משתמשים ב-Claude Code בדיוק אותו הדבר בשבוע הראשון ובחודש השישי. הכלי לא משתפר לבד – והם לא מוצאים זמן לשפר את הדרך שבה הם עובדים איתו. התוצאה: אותן טעויות חוזרות על עצמן, אותם רגעים שבהם צריך לעצור את הסוכן ולתקן אותו ידנית.

**הרעיון המרכזי הוא פשוט:** אם משפרים את התהליך באחוזים בודדים כל יום, האפקט המצטבר על פני שבועות הופך לעצום. הנקודה החשובה היא שזה לא חייב לקרות ידנית – אפשר להפוך את זה לתהליך אוטומטי לגמרי.

![תרשים המתאר איך להפוך את Claude Code לסוכן שמשתפר בעצמו](/images/posts/claude-learning.png)

## 1. הקימו "סוכן שמבקר את עצמו" בלילה

הטכניקה המרכזית: יוצרים skill מותאם אישית (אפשר לקרוא לו למשל `review-past-performance`) שמריצים אוטומטית כל לילה, בלי מעורבות אנושית.

### איך זה עובד?

• ה-skill מנחה את Claude לעבור על האינטראקציות מה-24 שעות האחרונות

• הוא מזהה חוסר יעילות, קריאות כלים מיותרות וטעויות חוזרות

• הוא בונה תוכנית: איך לייעל את העבודה גם בתוך הריפו הנוכחי וגם בין ריפואים שונים

• התובנות מתורגמות לשינויים קונקרטיים: תוספות ל-`CLAUDE.md` או קובצי קונפיגורציה דומים, בניית skills חדשים למשימות חוזרות, ויצירת סקריפטים (pre-commit hooks, סקריפטי בדיקה) שמונעים את הטעות מלכתחילה

בפועל אפשר פשוט ליצור skill בתוך Claude Code עם פרומפט שנראה בערך כך:

```
Review my last interactions with Claude Code from the last 24 hours. 
Look for any problems that I encountered, things that weren't working 
efficiently, and unnecessary tool calling. Look for common mistakes 
Claude Code was doing and other things that can be optimized. 
Look thoroughly through all conversations and make a plan for how we 
can optimize our flow in the future, both within each repository and 
cross-repositories. Also look for insights that would be useful for the 
coding agent to know beforehand, both before entering a repository and 
when working in multiple repositories at the same time.
```

### למה זה טוב?

הסוכן הופך להיות מותאם אישית לסטאק הטכנולוגי ולהעדפות שלכם – בלי שתצטרכו לכוונן אותו ידנית בכל פעם. פחות טעויות חוזרות, יותר הקשבה להעדפות שכבר הסברתם פעם אחת.

**למי מתאים:** מי שמריץ Claude Code על אותם ריפואים שוב ושוב ומרגיש שהוא "שוכח" תובנות מסשן לסשן.

## 2. הריצו כמה סוכנים במקביל

השינוי השני הוא לא בסוכן עצמו, אלא באופן שבו אתם עובדים מולו. במקום סוכן אחד שרץ בטור, אפשר להריץ כמה סוכנים על משימות שונות בו-זמנית.

### איך זה עובד?

• מפצלים חלון טרמינל לכמה panes – pane אחד לכל ריפו או משימה

• דואגים שהמשימות באמת ניתנות להרצה במקביל, עם כמה שפחות תלות הדדית

• עוקבים אחרי כל הסוכנים ומתערבים רק כשצריך

### שימו לב

יש תקרה מעשית. מעבר לכמות מסוימת של סוכנים פעילים בו-זמנית, השליטה מתחילה להישחק – קשה לעקוב אחרי מה שכולם עושים. כדאי למצוא את הסף האישי שלכם ולא לחצות אותו רק כי אפשר טכנית.

**למי מתאים:** מי שעובד על כמה משימות עצמאיות שלא ממש תלויות זו בזו – רפקטורינג בקובץ אחד ובדיקות בקובץ אחר, למשל.

## 3. הפכו את הכיוון: הסוכן עובד, לא שואל

השינוי המנטלי המרכזי בגישה הזו: במקום שאתם תשאלו את הסוכן שאלות כל הזמן, מבקשים ממנו לעבוד בעצמאות כמה שיותר זמן – ולעצור ולשאול אתכם רק כשהוא באמת תקוע.

### איך זה עובד?

• כותבים את זה מפורשות בפרומפט או ב-`CLAUDE.md`: "עבוד בעצמאות עד כמה שאפשר, עצור ושאל רק כשאתה חייב"

• משלבים את זה עם יכולת של הסוכן לבדוק את העבודה של עצמו לפני שהוא מדווח שהיא מוכנה – כך הוא יכול להמשיך לרוץ בלי לחכות לאישור על כל צעד

• ככל שהאמון גדל, מרחיבים את היקף המשימות שמקבלות את הטיפול הזה

**למי מתאים:** מי שמרגיש שהוא "מנהל מיקרו" את הסוכן במקום לתת לו לעבוד.

## סיכום: זה תהליך הנדסי מתמשך, לא הגדרה חד-פעמית

השיפור האמיתי ב-Claude Code לא מגיע מהתקנה טובה יותר בהתחלה, אלא מהרגל של לצוד את צוואר הבקבוק הכי גדול כרגע – בין אם זו טעות חוזרת של הסוכן או חיכוך בתקשורת בין אתכם לביניו – ולטפל בו שיטתית. cron job לילי שמבקר את הביצועים, כמה סוכנים במקביל כשזה מתאים, וגישה של "עבוד קודם, שאל אחר כך" – שלושה שינויים שכל אחד מהם דורש השקעה חד-פעמית קטנה ומחזיר את עצמו כל יום מחדש.
