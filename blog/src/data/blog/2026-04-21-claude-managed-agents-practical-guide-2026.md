רוב האנשים שמנסים לבנות סוכן AI נתקעים באותם מקומות: לולאת Agent מסובכת, ניהול context, הרצת קוד בסנדבוקס, שמירת state לאורך זמן, ואבטחה סביב מפתחות וגישה ל-GitHub או למסדי נתונים. התוצאה היא מערכות שבקושי מחזיקות יום בלי שצריך "לטפל בשרת", במקום סוכן יציב שאפשר לסמוך עליו.

הבעיה האמיתית היא שהרבה צוותים מנסים לבנות הכול לבד: גם "המוח" (Claude), גם "הידיים" (כלים, קוד, sandboxes), וגם "היומן" של מה שקרה. **Claude Managed Agents** פותר את זה דרך הפרדה חכמה: שירות מנוהל שמנהל עבורכם את הסשנים, ה-harness וה-sandbox, עם ממשק יציב שיכול להשתנות מתחת לפני השטח ככל שהמודלים משתפרים — בלי לשבור לכם את האפליקציה.

## 1. Managed Agents — סוכן מנוהל עם מוח, ידיים ויומן מופרדים

**Claude Managed Agents** הוא שירות מנוהל בפלטפורמת Claude שמריץ עבורכם סוכנים ארוכי-טווח דרך כמה ממשקים ברורים: **Session** (יומן אירועים), **Harness** (הלופ שמנהל את Claude וה-tools), ו-**Sandbox** (סביבה להרצת קוד ועריכת קבצים). ההפרדה הזו לא מקרית — היא פותרת בעיה ישנה במדעי המחשב: איך מעצבים מערכת ל"תוכניות שעוד לא הומצאו".

### למה זה טוב?

• **הפרדה ברורה בין רכיבים**: "המוח" (Claude + harness), "הידיים" (כלים וסנדבוקסים) ו-"היומן" (session log) לא תלויים אחד בשני
• **אבולוציה חופשית**: אפשר להחליף/לעדכן harnesses ו-sandboxes בלי לשנות את ה-API סביבם
• **זיכרון מתמשך**: הסשן הוא לוג append-only של כל מה שקרה, שנשמר מחוץ ל-context window של Claude
• **אמינות production**: השירות מתוכנן ל-long-horizon tasks, עם טיפול בכשלים ואבטחה ברמה גבוהה

### הכי מתאים ל:

צוותים שרוצים לבנות אפליקציות Agent רציניות (קוד, מסמכים, אינטגרציות) בלי לנהל לבד את הארכיטקטורה המלאה. אם אתם צריכים סוכן שעובד ימים או שבועות על משימה אחת, זה בשבילכם.

[התחילו עם Claude Managed Agents](https://www.anthropic.com/engineering/managed-agents)

## 2. ה-Session — לוג אירועים חיצוני שפותר את בעיית ה-Context

ב-**Managed Agents**, ה-Session הוא האובייקט שמכיל את כל האירועים: הודעות, קריאות כלים, תוצאות, שגיאות ועוד — **מחוץ לחלון ה-context** של Claude. המוח יכול "לשאוב" ממנו קטעים לפי הצורך באמצעות `getEvents()`. זה אומר שהיסטוריה מלאה נשמרת תמיד, גם כשהמודל "שוכח" משהו כי הקונטקסט התמלא.

### למה זה משנה?

• **אי-אובדן מידע**: כל מה שהסוכן עשה נשמר ביומן מתמשך, גם אחרי שה-harness התרסק או הוחלף
• **דיבאג פשוט**: אפשר "לגלול אחורה" כמה אירועים כדי להבין מה הוביל לפעולה מסוימת
• **אופטימיזציה של context**: אפשר לעשות טרנספורמציות על האירועים לפני שמכניסים אותם שוב ל-context — למשל סיכום או ארגון מחדש
• **שליטה מלאה**: ההחלטות איזה context להחזיר ל-Claude נמצאות ב-harness, אבל המידע עצמו תמיד נשמר

### איך זה עובד בפועל:

נניח שאתם בונים סוכן שעובד על פרויקט תכנות שלם במשך שבוע. הסשן שומר את כל השיחות, ה-commits, השגיאות והתיקונים. כשה-context מתמלא, ה-harness יכול לבקש מהסשן רק את ה-10 האירועים האחרונים + סיכום של מה שהיה לפני כן. Claude לא "רואה" את ההיסטוריה המלאה, אבל שום דבר לא אבד.

## 3. ה-Harness — הלוגיקה שמנהלת את הלולאה (בלי להיתקע על תשתיות)

ה-**Harness** הוא הלופ שמריץ את Claude, קורא ל-tools, כותב אירועים לסשן, ומטפל בשגיאות. Anthropic בנתה "meta-harness" שיכול להכיל harnesses שונים (למשל **Claude Code**, harness ייעודי לעבודה על קוד, או harnessים מותאמים אישית לפי משימות). ההבדל הגדול: ה-harness עצמו הפך ל-"cattle" במקום "pet".

### מה זה אומר בפועל?

• **עמידות לכשלים**: אם harness נופל, אפשר להרים חדש עם `wake(sessionId)` ולהמשיך מאותו לוג — אין "שרת יקר" שאסור לאבד
• **גמישות תשתיתית**: הוא לא חייב לחיות באותו container כמו הקוד — הוא קורא ל-sandbox כמו לכל tool חיצוני: `execute(name, input)`
• **ניהול context בידיים שלכם**: כל ניהול ה-context (summaries, trimming, prompt engineering) מתרחש ב-harness ולא "נצרב" ב-session
• **ארכיטקטורה מבוזרת**: אפשר להריץ כמה "מוחות" שונים (many brains) על אותו סט ידיים (many hands) לפי הצורך

### הכי מתאים ל:

מפתחים שרוצים לכתוב לוגיקת Agent משלהם, אבל ליהנות משירות מנוהל שמטפל ב-restart, שגיאות ו-scaling. אם אתם רוצים שליטה מלאה על האסטרטגיה של ה-agent אבל לא להמציא את הגלגל — זה הפתרון.

## 4. Sandboxes ו-Tools — הידיים: הרצת קוד, גישה ל-Git ו-MCP

ה-"ידיים" של הסוכן הן **sandboxes** וכלים: סביבות הרצה לקוד (containers) ו-**MCP servers** שמתחברים לשירותים חיצוניים. הם נצרכים דרך ממשק אחיד: `execute(name, input) → string`. זה אומר שה-harness לא צריך לדעת אם "היד" היא container, קריאת API, או שירות פנימי — הכול נראה זהה.

### למה זה טוב?

• **גם ה-sandboxes הם cattle**: אם container נופל, ה-harness רואה error ויכול להפעיל provisioning של container חדש אוטומטית
• **אבטחה מובנית**: ב-Git, הטוקן קשור לריפו — הסנדבוקס מקבל remote מוגדר מראש, אבל ה-agent לא נוגע בטוקן עצמו
• **שכבת Proxy לסודות**: ב-MCP, הטוקנים מאוחסנים ב-vault חיצוני; Proxy מיוחד שולף אותם ומבצע את הקריאה, בלי לחשוף אותם ל-Claude או ל-harness
• **הכללה מלאה**: ה-harness לא צריך לדעת מה הסנדבוקס עושה בפנים — רק שולח input ומקבל output

### דוגמה מהחיים:

נניח שאתם רוצים שהסוכן ידחוף קוד ל-GitHub ויעדכן טאסק ב-Jira. שני הכלים האלה דורשים OAuth tokens. במקום לתת לסנדבוקס גישה ישירה לטוקנים, אתם מגדירים MCP proxy שמבצע את הקריאות עבור Claude. גם אם מישהו ינסה prompt injection כדי "לגנוב" את הטוקן — הוא לא נמצא בסביבת הריצה בכלל.

## 5. אבטחה, סקיילינג ו-Latency — למה ההפרדה הזו משנה

הפרדת המוח, הידיים והסשן פתרה לאנתרופיק בעיות ותיקות: "שרתים שהם חיות מחמד", דליפת טוקנים מהסנדבוקס, ו-latency גבוה עד לטוקן ראשון (**TTFT** — Time To First Token). התוצאה: מערכת שמסתגלת טוב יותר לעומסים, מגיבה מהר יותר, ומאובטחת יותר.

### מה השתנה?

• **אמינות גבוהה**: אם container נתקע, לא מאבדים את הסשן — רק מרימים סביבת עבודה חדשה
• **אבטחה רב-שכבתית**: הטוקנים (Git, OAuth) לא נגישים מהסנדבוקס שבו רץ הקוד שנוצר על-ידי Claude, כך ש-prompt injection לא יכול "לגנוב" אותם
• **מהירות משופרת**: המוח לא מחכה ל-container אם לא צריך אותו — inference מתחיל מיד, וה-hands מוקצים רק בקריאה ל-tool
• **ביצועים מדידים**: ארכיטקטורת "many brains, many hands" הורידה את p50 של TTFT בערך ב-**60%** ואת p95 ביותר מ-**90%**

### הכי מתאים ל:

מוצרים שצריכים הן חוויית משתמש מהירה (זמן לתגובה ראשונה) והן אמינות/אבטחה בסקייל גבוה לאורך זמן. אם אתם בונים כלי פנימי שצוות שלם משתמש בו כל יום, או מוצר SaaS שלקוחות משלמים עליו — זה קריטי.

## 6. טבלת מטרה | כלי מומלץ

| **מטרה** | **כלי / רכיב מומלץ** |
|-----------|----------------------|
| להרים סוכן ארוך-טווח מנוהל | Claude Managed Agents |
| לשמור היסטוריית עבודה מלאה מחוץ ל-context | Session log + `getEvents()` |
| לנהל לוגיקת Agent, context ו-tools | Harness גמיש (meta-harness של Anthropic) |
| להריץ קוד ולגשת למשאבים חיצוניים | Sandboxes + MCP tools דרך `execute()` |
| להבטיח אבטחה ו-scaling לאורך זמן | הפרדת brain / hands / session + vaults |

## זרימת עבודה מומלצת: איך להתחיל עם Managed Agents

1. **הגדרת Agent**: מגדירים Managed Agent בקונסולת Claude, עם הכלים שהוא צריך — sandboxes להרצת קוד, MCP לחיבור ל-Git/DB/שירותים פנימיים, וה-session כבסיס הזיכרון שלו
2. **בחירת Harness**: בוחרים או מגדירים harness שמתאים למשימה (למשל מבוסס Claude Code או harness ייעודי), ודואגים שהוא משתמש ב-session log כדי לנהל context במקום "לחתוך" מידע לצמיתות
3. **חיבור API**: מחברים את האפליקציה שלכם ל-Managed Agents דרך ה-API — יוצרים sessions, מאזינים לאירועים, ומאפשרים לסוכן לקרוא לכלי ה-sandbox/MCP דרך הממשקים הסטנדרטיים
4. **אבטחת סודות**: מוודאים שהטוקנים וכל האישורים נשמרים מחוץ לסנדבוקס (vaults, proxies), כך שה-agent רואה רק interfaces ולא סודות גולמיים
5. **ניטור ואופטימיזציה**: ב-production, מנטרים TTFT, כמות אירועים ואחוזי כישלון tools, ומכוונים את ה-harness (context strategies, תזמון, retries) בלי לשנות את session או sandboxes

## שורה תחתונה

**Managed Agents** נותן לכם ארכיטקטורה מוכנה ל-Agents רציניים: מוח חכם, ידיים גמישות, וזיכרון יציב — בלי שתצטרכו להמציא לבד איך מפצלים ביניהם בצורה מאובטחת וסקיילבילית. זה לא עוד כלי AI שמבטיח "מהפכה", אלא פתרון מהנדסי מוצק לבעיה שכל מי שניסה לבנות סוכן ארוך-טווח פגש: איך לא להפוך את השרת שלכם ל-"חיית מחמד" שאסור שתיפול.

אם אתם חושבים רצינית על Agents בפרודקשן — למשל אוטומציה של תהליכי קוד, בדיקות, או עבודה עם מסמכים — כדאי לכם להתחיל מהיסודות הנכונים. הארכיטקטורה של Managed Agents תחסוך לכם חודשי פיתוח ותפתור בעיות אבטחה וסקיילינג שרוב הצוותים פוגשים רק אחרי שהמערכת כבר בייצור.

אם חשוב לך להישאר מעודכן בכלים ובשיטות עבודה אמיתיות מהשטח, תצטרף לקהילה **AI World** — שם מדברים פתוח על איך עובדים חכם יותר עם AI בפיננסים ובעסקים.
